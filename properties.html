Hallazgos Clave

Patrones que mas penalizan una subida limpia a JDK 17.

JAXB fuera del JDK DeltaSSIs, DeltaSSIsFX y DeltaSSIProject rompen compilacion si no se anaden dependencias JAXB externas.
Clases internas `com.sun.*` ProjectSQL usa `CachedRowSetImpl`, una API interna fragil para JDK 17.
Stacks legacy de framework La familia Project* arrastra Spring 3, Hibernate 3 y Log4j 1: el coste esta en la plataforma, no en la sintaxis Java.
Integracion sensible servicesRDR mezcla sockets, comandos externos, validacion XML y JMS. Es un proyecto de alta necesidad de prueba funcional.
Muy pocos tests Solo 9 proyectos tienen tests detectables. El mayor riesgo no es migrar; es no saber si rompes comportamiento.

Lectura operativa

Por que estos workflows entran en pruebas JDK 17.

CommandLine dinamico Muchos pasos `CALL JAVA` ejecutan variables como `cmd`, `script` o `comando`; el JAR real se resuelve en runtime.
Eventos como entrada real Los eventos `GenericEvent` y `OnJMSMessage` pueden activar workflows que lanzan Java aunque el proyecto no aparezca en properties.
Prueba obligatoria Para JDK 17 hay que capturar el comando final resuelto y ejecutar smoke con datos o mensajes anonimizados.

Division Recomendada de Subidas

Tres ventanas de entrega y cuatro bloques logicos, separando `rdrRules` por su uso transversal.

12/09 Bloque 1 de bajo a medio riesgo. Objetivo: asegurar primeras subidas con proyectos Maven acotados y build estable.
03/10 Bloque exclusivo de `rdrRules`. Va separado porque es transversal y tiene mayor impacto funcional dentro del ecosistema.
07/11 Dos corrientes paralelas: bloque 3 para incompatibilidades tecnicas localizadas y bloque 4 para legacy/plataforma de mayor riesgo.

Entrega 12/09 · Primera ola
12/09

`rdrRules` debe viajar solo. Aunque no sea el score maximo del inventario, funcionalmente es de alto impacto porque se reutiliza mas dentro del proyecto y arrastra mas riesgo de regresion operativa.
